# PRD: TabTypist v1

**Status:** needs-triage
**Scope:** macOS, Local mode, English + Amharic
**References:** [CONTEXT.md](../../CONTEXT.md), [ADR-0001](../adr/0001-rust-core-with-native-ui-shells.md), [ADR-0002](../adr/0002-default-on-in-messaging-apps.md), [ADR-0003](../adr/0003-hosted-mode-shape-deferred-to-v2.md), [ADR-0004](../adr/0004-functional-source-license.md)

## Problem Statement

People who type all day in apps across their machine want AI assistance that **feels like part of typing**, not a separate tool they switch into. CoTypist solved this on macOS but is closed-source and English-centric. Cotabby is open-source but English-only and macOS-only forever (Swift-locked architecture).

Two underserved audiences fall out of this gap:

- **Non-English typists, Amharic first.** Existing tools either don't support their script at all or do so via tokenizers that fragment every word into 10+ tokens, making inline completion painfully slow and quality-poor.
- **Users on hardware that can't run a capable local model.** Low-end Windows machines and older Macs are excluded from the existing landscape, which assumes Apple Silicon + 16GB+ RAM.

TabTypist v1 addresses the first audience directly (macOS, Amharic-capable local model) and lays the architectural foundation for the second (Rust core that the v2 Windows port reuses; Hosted mode that activates when local hardware can't keep up).

## Solution

A macOS menu bar app that watches what the user is typing into any text field and offers **inline ghost-text Completions** suggesting how their sentence might continue. The user accepts with **Tab**, dismisses with **Escape**, and otherwise keeps typing — TabTypist gets out of the way.

Two languages ship at v1: **English** (default model: Qwen 2.5 1.5B, 4-bit, ~900MB) and **Amharic** (default model: EthioNLP Amharic-llama 7B base, 4-bit, ~4GB, pending native-speaker eval before final commit). Users choose languages at onboarding; an "Advanced" disclosure lets power users override the default model. All inference is **Local mode** — fully on-device, no network calls — and that is enforced architecturally, not just by configuration. Hosted mode and the Windows port are explicitly v2.

Trust is the foundation: the app honors a three-tier **Exclusion list** (secure-text fields always off; password managers and major banking apps default-off; messaging apps default-on with a first-activation toast); the menu bar icon shows a visible **paused affordance** in any app where TabTypist is not active; permissions are requested in two phases (Accessibility before first use, Input Monitoring only after the user has accepted their first Completion); telemetry is opt-in, off by default, and never includes any Completion or field text.

The license is **Functional Source License 1.1** with Apache 2.0 as the future license at 2 years (see ADR-0004). Distribution is via direct DMG download (notarized in CI) and Homebrew cask, with Sparkle 2.x auto-update.

## User Stories

1. As a writer who works across Mail, Notes, Slack, and a browser textarea all day, I want ghost-text completions to appear as I type in any of those apps, so that I don't have to switch into a separate tool to get AI help.
2. As an Amharic typist, I want to download an Amharic-capable model during onboarding, so that I can use TabTypist in my primary writing language without configuring anything technical.
3. As a bilingual user, I want to select both English and Amharic at onboarding, so that TabTypist can offer Completions in whichever language I'm currently typing without me toggling anything.
4. As a privacy-conscious user, I want to confirm visually that TabTypist runs entirely on my machine, so that I trust the product to read what I type.
5. As a user starting a sentence, I want a Completion to appear within a few hundred milliseconds of when I pause typing, so that the suggestion arrives in the rhythm of my writing rather than after I've already moved on.
6. As a user accepting a suggestion, I want to press Tab and have the ghost text become real text in the field, so that the gesture is consistent with how every other inline-completion tool I've used works.
7. As a user rejecting a suggestion, I want to press Escape and have the ghost text vanish, so that I can dismiss without committing.
8. As a user typing past a stale suggestion, I want the Completion to vanish on its own when what I'm typing no longer matches, so that I never see ghost text that contradicts what I just wrote.
9. As a developer using Tab for indentation in VS Code, I want Tab to behave normally when no Completion is showing, so that TabTypist doesn't break my normal Tab usage.
10. As a user mid-sentence, I want Completions to use the text **after** my cursor as well as before, so that mid-sentence rewrites and inserts produce sensible suggestions.
11. As a user during onboarding, I want a welcoming, polished first-launch experience that explains what TabTypist does in 30 seconds, so that I understand the product before I'm asked to grant permissions.
12. As a user during onboarding, I want to be asked for Accessibility permission with honest copy explaining what it's used for, so that I can make an informed decision.
13. As a user mid-onboarding, I want the model download to start in the background while I keep reading the explanation, so that I'm not staring at a progress bar.
14. As a user, I want to be asked for Input Monitoring permission **after** I've already seen TabTypist work once, so that I understand the value before granting the scarier permission.
15. As a user typing in 1Password, I want TabTypist to silently not run, so that my passwords aren't read by any process I didn't explicitly authorize there.
16. As a user typing in a banking app, I want TabTypist to be off by default, so that financial entries are never processed by an AI tool I forgot was running.
17. As a user typing in a system password prompt or any secure-text field, I want TabTypist to be unconditionally off, so that the OS-level signal about secret entry is always respected.
18. As a user typing in Signal for the first time, I want a one-time toast informing me that TabTypist is active here and offering to disable it, so that I make an informed choice rather than being silently observed.
19. As a user who decided to disable TabTypist in Slack, I want to remember that decision across app restarts, so that I don't re-grant access I revoked.
20. As a user glancing at the menu bar, I want a visible paused state when TabTypist is off in the focused app, so that I always know whether it's listening.
21. As a power user, I want to override the default model with one of my choice during onboarding, so that I can opt into a larger or smaller model than the default.
22. As a power user, I want to change my model choice from settings later, so that I don't have to reinstall to experiment.
23. As a user opening a fresh install, I want to see clearly how much disk a model download will take **before** I trigger the download, so that I can decide consciously.
24. As a user with a flaky connection, I want a paused model download to resume from where it left off, so that I don't have to start over after a network hiccup.
25. As a user installing TabTypist, I want the downloaded model file to be checksum-verified before being trusted, so that a corrupt or tampered download never gets loaded.
26. As a user concerned about telemetry, I want telemetry to be off by default with a clear opt-in checkbox during onboarding, so that I'm not surprised later by data leaving my machine.
27. As a user who opted into telemetry, I want to be able to revoke consent from settings at any time, so that I can change my mind.
28. As a user, I want to know exactly what telemetry collects (and what it never collects), so that I can audit the claim.
29. As a user, I want auto-updates to be code-signed and notarized, so that I trust each update as much as I trusted the original install.
30. As a developer evaluating TabTypist, I want to be able to read the source code, so that I can verify the privacy claims independently.
31. As a Homebrew user, I want to `brew install --cask tabtypist`, so that the install fits into my usual tooling.
32. As a user, I want the onboarding flow to feel as polished as the rest of the app, so that my first impression matches the product's positioning.
33. As a user uninstalling TabTypist, I want the uninstall to remove all model files, settings, and any cached data, so that I'm not left with stale data on disk.

## Implementation Decisions

### Architecture

- **Rust core** owns all platform-agnostic logic: completion orchestration, model loading, language routing, exclusion enforcement, settings persistence, downloader, telemetry. (ADR-0001.)
- **Swift AX sidecar** runs as a child process spawned by the Rust core, communicating over JSON-RPC on stdin/stdout. It owns AXUIElement reads, caret-rect reporting, and global Tab/Escape capture.
- **Swift SwiftUI shell** hosts the menu bar UI, settings panel, onboarding flow, and overlay renderer.
- **MLX is deferred** to v1.5 or later. v1 ships **llama.cpp** linked into the Rust core via `llama-cpp-2`. The Model Runtime is structured behind a `Completer` trait so MLX and Hosted implementations can slot in later without touching callers. (ADR-0001.)

### Module set

1. **Completion Engine** (Rust) — deep. Owns debounce (~150ms post-keystroke), in-flight cancellation, divergent-typing dismissal, sentence-boundary truncation. Interface accepts a `ContextProvider` and emits a stream of Completion events with cancellation tokens.
2. **Model Runtime** (Rust) — deep. `Completer` trait; v1 has `LlamaCppCompleter`. Loads a model file, runs decode with stop-tokens and max-token cap (~25).
3. **Language Router** (Rust) — deep, pure. Maps (script-detected from prefix, user settings) → which loaded `Completer` to call.
4. **Exclusion Engine** (Rust) — deep, pure logic + signed remote config. Three-tier verdict: always-off / default-off-user-can-enable / default-on-user-can-disable. Honors the OS secure-text-field signal as always-off.
5. **Model Catalog & Downloader** (Rust) — deep. Lists available models (English default, Amharic default, advanced overrides), manages resumable downloads, verifies checksum + signature, stores under a known on-disk layout, exposes progress events.
6. **Settings Store** (Rust) — deep, schema-versioned. Persists selected languages, model overrides, user-edited exclusion list, telemetry consent, mode (always Local at v1).
7. **Telemetry Client** (Rust) — deep. Enforces "no transmit before consent" invariant. Anonymous event batching + crash-report sender. Random install ID resettable from settings.
8. **AX Sidecar** (Swift) — shallow. JSON-RPC over stdio. Three message families: focus events, context reads (prefix/suffix/caret), key captures (Tab/Escape).
9. **Overlay Renderer** (Swift) — shallow. Borderless `NSPanel` positioned at the reported caret rect, ghost-text styled to approximate host-app font.
10. **Menu Bar UI** (SwiftUI) — shallow. Status icon with active/paused affordance, settings panel, exclusion-list editor, current-app "Disable" toggle.
11. **Onboarding Flow** (SwiftUI + Rust state machine) — mixed. Two-phase progressive sequence per ADR-0002 timing: Welcome → Language Selection → Accessibility Permission → Background Download Interstitial → First-Use Tooltip → Input Monitoring on First Accept → Exclusion-list awareness on first paused-app encounter.
12. **Updater** (Swift, Sparkle 2.x) — shallow. EdDSA-signed feed; beta and stable channels.

### Domain language

The PRD uses the canonical terms from `CONTEXT.md`: **Completion**, **Acceptance**, **Dismissal**, **Trigger**, **Local mode**, **Default model**, **Model override**, **Exclusion list**. Do not introduce synonyms in code or copy.

### Permissions

- **Accessibility** — required, requested at onboarding step 3.
- **Input Monitoring** — required, requested after the user's first **Acceptance** (deferred from initial onboarding to soften the permission load).
- **Screen Recording** — not requested at v1. Used only by the deferred screenshot-OCR context source.

### Exclusion list policy

- Always-off: `AXSecureTextField`, OS keychain/password prompts, any field the OS flags as password-style.
- Default-off, user can re-enable: 1Password, Bitwarden, Dashlane, Keeper, Apple Passwords, and a curated list of major US/EU native banking apps (by bundle ID).
- Default-on, user can disable: everything else, including Signal, Telegram, WhatsApp, iMessage. A one-time activation toast appears the first time TabTypist activates in each of those four messaging apps (ADR-0002).
- The exclusion list is delivered as a signed, remotely-updatable config so the app can react to incidents without a release.

### Licensing

- FSL 1.1 with Apache 2.0 future license, 2-year conversion (ADR-0004). The repo `LICENSE` file uses the canonical FSL text from fsl.software.
- No CLA at v1. Outside contributions are accepted under FSL terms; the future-license clause carries those contributions into Apache 2.0 on the same schedule.

### Distribution

- Apple Developer ID Application certificate.
- GitHub Actions builds and notarizes via `notarytool`.
- Direct DMG download from the TabTypist website + Homebrew cask formula.
- Sparkle 2.x with EdDSA-signed appcast for auto-update; beta and stable channels available from v1.

## Testing Decisions

**What makes a good test here:** test external behavior, not implementation details. For Rust modules, that means asserting against the module's public interface and observable state changes, never against internal helpers. For trust-critical invariants (Exclusion Engine, Telemetry Client), include explicit "negative" tests that fail loudly if the invariant is violated — these are the tests that matter most for protecting users.

**Modules with tests at v1:** the trust-critical and behavior-critical four.

- **Completion Engine (#1).** Unit tests against the engine's public interface using a stub `Completer` and a stub `ContextProvider`. Cover: debounce coalesces a burst of triggers into one request; in-flight cancellation actually cancels (stub asserts cancel was called); divergent typing after a Completion is produced causes silent dismissal; sentence-boundary truncation stops at `.!?\n` and at the 25-token cap.
- **Exclusion Engine (#4).** Unit tests over the verdict function for representative apps + field types. Cover: secure-text-field always returns off regardless of user config; password-manager bundle IDs return off by default; messaging app bundle IDs return on by default with the activation-toast flag set on first encounter; user overrides persist and take precedence within their tier; remote config signature validation rejects tampered configs.
- **Model Catalog & Downloader (#5).** Unit tests against the downloader's interface with an HTTP test double. Cover: resumable downloads pick up from byte offsets; corrupted bytes fail checksum verification and are not promoted to "installed"; an untrusted signature on a model file aborts before any code is loaded; partial files don't pollute the installed-models directory.
- **Telemetry Client (#7).** Unit tests against the client's public interface with a network double. Cover: no network call is made before consent is recorded as true (the trust-critical invariant); revoking consent halts in-flight batches; no event contains any field text, completion text, or PII beyond app version + OS version + model ID + install ID; install ID reset produces a new ID on next event.

**Modules not unit-tested at v1:** Model Runtime (#2) is exercised end-to-end via the Completion Engine tests with the real llama.cpp adapter behind a feature flag; Language Router (#3) and Settings Store (#6) get tested incidentally through the engines that depend on them and via a small number of direct tests as needed during development. AX Sidecar (#8), Overlay Renderer (#9), Menu Bar UI (#10), Onboarding Flow (#11), and Updater (#12) are integration-tested manually because they are UI- or system-API-bound.

**Prior art:** TabTypist has no codebase yet, so there is no prior art to follow. Establishing the testing patterns is part of the early implementation work — first Rust module written should set the convention for the rest.

## Out of Scope

- **Hosted mode.** The v2 design is captured in ADR-0003 and explicitly does not ship at v1.
- **Windows.** Architecturally enabled by the Rust core, but the Windows shell, UIA sidecar, and any Windows-specific behavior are v2.
- **MLX runtime.** Deferred to v1.5 or later (ADR-0001).
- **Screenshot OCR for AX-invisible apps (Google Docs in canvas mode, etc.).** Deferred to v1.5 or later. The `ContextProvider` interface is shaped today so adding it is additive.
- **Style memory / personalization.** v2+.
- **Partial-accept by word (Option-Tab or similar).** v1.1.
- **Tab-to-cycle alternative completions.** Not on the roadmap.
- **Multi-completion candidates.** v1 generates one Completion per Trigger.
- **Mac App Store distribution.** Impossible for this product class due to sandbox restrictions on Accessibility + Input Monitoring + system-wide overlay windows.
- **CLA for outside contributions.** Skipped at v1; revisitable if relicensing becomes desirable later, with the constraint that any pre-CLA contributions remain under FSL-then-Apache forever.
- **Premium frontier-quality completions for users who could run local but want more.** Explicitly rejected as a Hosted-mode job in ADR-0003.

## Further Notes

- **Amharic model commit is gated on a native-speaker eval.** Before EthioNLP Amharic-llama 7B is locked as the Amharic default, build a 50-sentence eval set (news, casual, formal registers) and have a native speaker rate Completions blind against the iocuydi 3.78B fallback. If neither hits acceptable quality, revisit the model choice; do not ship an Amharic experience that embarrasses the project.
- **Onboarding is a v1-blocking commitment.** The visual UX must be polished — running the `/shape` skill before implementation is strongly recommended so that the design isn't an afterthought.
- **The exclusion-list signed remote config is a v1 capability.** The signing infrastructure, the config-distribution endpoint, and the in-app verification logic must all ship at v1. Without remote update, incidents require an app release to respond to.
- **First-activation toast in messaging apps is non-negotiable** per ADR-0002. It is the privacy mitigation that makes the default-on policy defensible.
- **The Rust↔Swift FFI boundary will be the most expensive seam to build.** UniFFI is the recommended starting point; budget time for it to feel awkward before it feels normal.
